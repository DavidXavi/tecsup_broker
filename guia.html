1. Panorama general

Pensa en un cartero en un edificio de oficinas. Llegan paquetes de distintas empresas (los publishers). El cartero no los entrega a cualquiera — solo a las oficinas que se anotaron para recibir cierto tipo de paquete (los subscribers).

Ese cartero es el broker. Su unica responsabilidad es recibir mensajes de A y entregarlos a B, C y D — sin que A sepa quienes son B, C o D. Eso se llama desacoplamiento, y es una de las ideas mas poderosas de la arquitectura de software.

Tres protagonistas:
Rol¿Qué hace?Analogía
Publisher Envia mensajes a un topico La empresa que manda un paquete
Broker Recibe, persiste y enruta mensajes El cartero inteligente
Subscriber Recibe mensajes de topicos a los que se suscribio La oficina que recibe el paquete

2. El protocolo — el "idioma" que todos hablan

Para que el cartero entienda los paquetes, todos deben usar el mismo formato. Eso es el protocolo: las reglas de como se escribe un mensaje en bytes para que el otro lado lo entienda.

Nuestro protocolo tiene dos capas:

2.1 Formato fisico (wire format)

┌───────────────┬──────────────────────────┐
│   8 bytes     │       N bytes            │
│   00000067    │  {"cmd":"PUBLISH",       │
│  (longitud)   │   "tpc":"clima",         │
│               │   "pld":{"temp":25}}     │
└───────────────┴──────────────────────────┘

Los primeros 8 bytes dicen "el mensaje mide 67 caracteres". El que lee sabe exactamente cuantos bytes esperar, sin adivinar. Esto es lo mismo que hacen HTTP (Content-Length) y muchisimos protocolos reales.

2.2 Comandos disponibles

Comando¿Quién lo envía?Significado
CONNECTCliente"Hola broker, quiero conectarme"
PUBLISHPublisher"Entrega esto en el topico X"
SUBSCRIBESubscriber"Quiero recibir todo lo del topico X"
UNSUBSCRIBESubscriber"Ya no quiero recibir del topico X"
MESSAGEBroker"Te entrego este mensaje que pediste"
OKBroker"Recibido, todo bien"
ERRORBroker"Algo salio mal"
HEARTBEATCliente"Sigo vivo, no me desconectes"
DISCONNECTCliente"Chau, me voy"
Dato clave: El archivo protocol.py tiene solo 60 lineas. Define el Frame (el sobre) y los metodos encode() y decode() para convertir entre objetos Python y bytes en la red. Eso es todo el protocolo.

3. El servidor — el corazon del broker

El servidor es el cartero. Vive en broker/server.py y hace tres cosas fundamentales:

3.1 Aceptar conexiones

Cuando un cliente nuevo golpea la puerta (socket.accept()), el servidor le da la bienvenida con un OK que contiene su client_id — un identificador unico de 8 caracteres como abc12345. Desde ese momento, el cliente esta "adentro".

3.2 El event loop — un solo hilo para miles de conexiones

El servidor usa selectors.DefaultSelector(), que es una herramienta del sistema operativo para decir "avisame cuando algun socket tenga datos listos para leer". En lugar de crear un hilo por cada cliente conectado, el servidor hace: 

while True:
    events = self.sel.select(timeout=1.0)
    for key, mask in events:
        if key.data is None:
            self._accept(key.fileobj)       # nuevo cliente
        else:
            self._handle_client(key.fileobj) # datos de un cliente existente

Esto escala muchisimo mejor que un hilo por conexion. Es el mismo patron que usa Nginx, Node.js y Redis.

3.3 El router — ?a quien le entrego esto?

El corazon del enrutamiento es un diccionario: 

self.subscriptions = {
    "clima":   {"sub_abc", "sub_def"},
    "bolsa":   {"sub_ghi"},
    "deporte": {"sub_abc", "sub_jkl", "sub_mno"},
}

Cuando llega un PUBLISH al topico "clima", el servidor busca en ese diccionario y reenvia el mensaje a sub_abc y sub_def. Operacion O(1).

4. El cliente — una sola clase para publicar Y suscribir

Esta es una decision de diseno importante: en lugar de tener un PublisherClient y un SubscriberClient separados, tenemos una unica clase BrokerClient que sabe hacer ambas cosas.

¿Por qué una sola clase?

Porque en la vida real, un servicio muchas veces publica Y consume. Un microservicio de pedidos publica "pedido_creado" y se suscribe a "pago_confirmado". Una sola conexion TCP al broker soporta ambos roles — ahorramos recursos y simplificamos el codigo.

¿Cómo se usa?

from broker import BrokerClient

# Cualquier cliente puede publicar, suscribirse, o ambos
client = BrokerClient()
client.connect()

# Como suscriptor
def cuando_llegue_mensaje(topic, payload, msg_id, timestamp):
    print(f"Recibi en {topic}: {payload}")

client.on_message("noticias", cuando_llegue_mensaje)
client.subscribe("noticias")

# Como publicador (en el mismo cliente!)
client.publish("alertas", {"mensaje": "Servidor caido"})

El hilo de escucha

Al conectarse, el cliente lanza un hilo en segundo plano que se dedica exclusivamente a leer del socket. Cuando llega un MESSAGE del broker, invoca los callbacks que registraste con on_message(). Asi no bloqueas el hilo principal — vos haces tu trabajo y los mensajes te llegan cuando el broker los envia.

Importante: un solo hilo lee del socket para evitar condiciones de carrera. Las operaciones de envio (publish, subscribe) usan un lock para no pisarse entre ellas.

5. Almacenamiento — donde queda la huella

Todo mensaje que pasa por el broker queda guardado en la carpeta broker_data/. Dos archivos simples, legibles con cualquier editor:

topics.json

[
  "noticias",
  "clima",
  "deportes"
]

Lista plana de todos los topicos que alguna vez se usaron.

messages.jsonl

{"message_id": "a1b2c3d4e5f6", "topic": "noticias", "payload": {"texto": "Hola mundo"}, "timestamp": "2026-07-01T14:30:00Z"}
{"message_id": "b2c3d4e5f6a7", "topic": "clima", "payload": {"temp": 25}, "timestamp": "2026-07-01T14:30:05Z"}

Una linea por mensaje — un append-only log. Nunca se borra, solo se agrega al final. Esto es exactamente el mismo principio que usa Kafka internamente: escritura secuencial, inmutable, ultra rapida.

Para el profe: El patron append-only log es uno de los conceptos mas importantes en sistemas distribuidos. Lo usan Kafka, PostgreSQL (WAL), SQLite (journal), y practicamente cualquier base de datos seria. Aca lo ves en su forma mas pura: un archivo de texto que solo crece.

6. Los ejemplos — publicador, suscriptor y demo

6.1 Publisher de ejemplo

Un programa simple que se conecta al broker y publica un mensaje cada 2 segundos en el topico "noticias".

from broker import BrokerClient

client = BrokerClient()
client.connect()

count = 0
while True:
    count += 1
    client.publish("noticias", {
        "numero": count,
        "contenido": f"Mensaje #{count}",
    })
    time.sleep(2.0)

Ejecucion: python -m examples.publisher [host] [puerto] [topico] [intervalo]

6.2 Subscriber de ejemplo

Se conecta, se suscribe a un topico, y muestra cada mensaje que recibe.

from broker import BrokerClient

def mostrar(topic, payload, msg_id, ts):
    print(f">> [{topic}] {payload}")

client = BrokerClient()
client.connect()
client.on_message("noticias", mostrar)
client.subscribe("noticias")

while True: pass  # el hilo de escucha hace el trabajo

Ejecucion: python -m examples.subscriber [host] [puerto] [topico]

9. Patrones de arquitectura en accion

Patrón¿Dónde se ve?¿Por qué lo usamos?
Cliente-Servidor Relacion Broker ↔ Clientes Un punto central que sirve a muchos — separacion clara de responsabilidades
Pub/Sub Enrutamiento por topicos Desacopla emisores de receptores — el que envia no sabe quien recibe
Event Loop selectors en el servidor Miles de conexiones con un solo hilo — eficiencia sin complejidad de threads
Protocolo con prefijo de longitud Frame.encode/decode El receptor sabe exactamente donde termina cada mensaje — sin ambiguedades
Append-only Log MessageLog Escritura inmutable secuencial — rapida, simple, auditable
Observer (callback) client.on_message() El suscriptor registra interes y es notificado cuando llega algo — inversion de control

10. De aca a produccion — ?que faltaria?

Lo que construiste es el motor del sistema. En produccion se le agrega carroceria, pero el motor no cambia:

Hoy (curso)Mañana (producción)¿Por qué?
1 solo broker Cluster con Raft/Paxos Si se cae el unico broker, chau sistema. Con 3 o 5 nodos, sobrevive.
Sin autenticacion TLS + JWT Cualquiera podria conectarse y publicar basura. En produccion, cada cliente se identifica.
JSON en texto Protobuf / Avro Esquema estricto + compresion binaria = menos bytes en la red, validacion automatica.
Sin garantia (at-most-once) ACK + reintentos + DLQ Si el suscriptor se cae, el mensaje se pierde. Con ACK, el broker reintenta hasta que confirme.
Suscriptores offline pierden mensajes Offset por suscriptor (como Kafka) Cada suscriptor recuerda "lei hasta el mensaje 57". Si se desconecta, al volver sigue desde el 58.
Archivos planos WAL + Snapshots Para millones de mensajes, necesitas compactacion periodica sin perder el historial.
Sin monitoreo API REST + dashboard En produccion necesitas saber cuantos mensajes por segundo, tasas de error, latencia.
